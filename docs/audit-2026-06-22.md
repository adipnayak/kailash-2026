Audited against: `https://adipnayak.github.io/kailash-2026/` (main, commit `2b7c921`).
Methodology: live `curl` checks + full source read of `app/src/**` on main branch + code analysis at mobile (375px) and desktop (1280px) widths. Preview server unavailable (running on a different project); all interactive checks performed via source-code analysis and live network requests.

## Critical (visible bugs hitting real users right now)

- [ ] **ThemeToggle button is 32x32 px -- below the 44 px minimum touch target.** File: `app/src/components/ThemeToggle.tsx` line 48 (`h-8 w-8`). On mobile it sits at the far right of the sticky nav with no padding expansion. A typical finger miss rate at 32 px is high; users sharing on WhatsApp use phones. Fix: change to `h-11 w-11` (44 px) or add `-m-1.5 p-1.5` padding trick to keep visual size but expand hit area.

- [ ] **Static HTML fallback lists wrong Prepare categories.** File: `app/index.html` line 116. Says "5-category preparation checklist (documents, packing, medicines, connectivity, fitness)". Actual categories in `app/src/lib/prep-data.ts` are: Passport, Flights, Medical, Connectivity, Packing. "Fitness" does not exist as a category; "documents" incorrectly collapses Passport and Flights into one. LLM crawlers and search engines without JS see the stale category list. This is the fallback that just shipped in PR #148.

- [ ] **Day 8 sleep altitude mismatch: static fallback says 4,760 m, `trip-data.ts` says 4,670 m.** File: `app/src/lib/trip-data.ts` line 650 (`altitude_sleep: 4670`) vs `app/index.html` line 91 ("Sleep 4,760 m"). Zuthulphuk camp is typically 4,760-4,780 m in published route guides, so the `trip-data.ts` value appears to be the error. The SPA (itinerary DayCard) shows the wrong 4,670 m value to all JS-enabled users. Day 9 in `trip-data.ts` has `altitude_sleep: 4760` which more accurately describes Zuthulphuk -- the Day 8 and Day 9 sleep altitudes appear swapped.

## Important (visible issues, not critical)

- [ ] **Nav inactive tabs are icon-only on every viewport.** File: `app/src/components/Nav.tsx` lines 43-47. When a tab is not active, its label is wrapped in `<span className="sr-only">`. At 375 px only one tab shows text at a time; the other three show icons only. `title` tooltip attribute helps desktop hover but does nothing on touch. WhatsApp group members sharing this link with family members may not recognise which tab does what. Consider always showing label text below the icon (as a small mono label) rather than hiding it entirely.

- [ ] **WeatherConfidence section fetches today's live weather but labels it with a "14% confidence" chip for the trip.** File: `app/src/components/WeatherConfidence.tsx` lines 97-104, 326, 375-385. The component fetches `forecast_days=2` from Open-Meteo -- this returns today's (22 Jun) weather at Kathmandu/Lhasa/Mansarovar. It then renders this under a heading "14% confidence forecast" which implies it is the trip-date forecast. It is not: the trip starts in 15 days, well outside the 16-day window for Tibet locations. The label is technically accurate (confidence = 14%) but the data shown (today's weather at those locations, not trip-date projections) is likely to confuse yatris. Consider showing StaticRanges (climatology) when confidence < 20% even if live data is available.

- [ ] **All 13 DayCards expand by default when the Itinerary tab is opened, causing 13 Leaflet maps to instantiate simultaneously.** File: `app/src/components/tabs/ItineraryTab.tsx` line 26-28 (`new Set(DAYS.map(d => d.day))`). Each card also lazy-loads `ItineraryDayMap` via `<Suspense>`. On a slow 4G connection (common in India and Nepal) this fires 13 simultaneous CartoDB tile fetches + up to 13 OSRM route fetches. The page may appear frozen for 2-4 seconds. Consider defaulting to only the first day expanded, or expanding lazily as the user scrolls.

- [ ] **ConfidenceBar in WeatherConfidence uses `rounded-full` for the progress bar track and fill, violating the `rounded-none` design rule.** File: `app/src/components/WeatherConfidence.tsx` lines 281-285. The bar uses `h-2 rounded-full bg-border` while every other border and shape in the SPA uses `rounded-none` (sharp corners). Minor visual inconsistency but visible on the Prepare tab.

- [ ] **Day 7 sleep altitude: static fallback says 4,890 m, `trip-data.ts` says 4,900 m (+10 m).** File: `app/src/lib/trip-data.ts` line 569 vs `app/index.html` line 92. A 10 m difference -- may not matter medically but the two surfaces disagree.

## Minor (polish, future)

- [ ] **ConnectivityRibbon status dots have `title` tooltip text but no interactive role.** File: `app/src/components/ConnectivityRibbon.tsx` line 121 (`<li ... title=...>`). The dots surface `title` attribute which works on desktop hover but is invisible on touch. They are `<li>` elements with no `role="button"` and no click handler, so keyboard users cannot access the tooltip text. If the per-day callout is informational only this is acceptable, but the `title` attribute creates a false affordance on desktop.

- [ ] **`bannerKey` value ('prep-window') is computed in journey state but never rendered.** File: `app/src/lib/journey-state.ts` line 88-95. The state engine computes `bannerKey: 'prep-window'` for 7-21 days before departure (current window). Neither `OverviewTab` nor `Hero` reads or renders it. Dead code creating some confusion, or a planned banner that was never wired up.

- [ ] **Middle dot separator (U+00B7 `·`) appears in rendered JSX strings in DayCard.** File: `app/src/components/DayCard.tsx` lines 337, 339, 702, 704 (`{' · '}`); also in `WeatherConfidence.tsx` line 385. The anti-AI rule bans em-dashes, en-dashes, smart quotes, and emojis -- middle dot is not in the list. But it is a non-ASCII separator that some brand guidelines and copy tools auto-convert. Noting for awareness; not a rule violation as written.

- [ ] **Day 7 trek distance inconsistency: timeline event shows "~12 km", static fallback shows "~18 km".** File: `app/src/lib/trip-data.ts` line 592 vs `app/index.html` line 92. The 18 km figure likely includes the road drive from Mansarovar to Darchen before the trek gate; 12 km is the actual trek-only distance (Yamadwar to Dirapuk). The DayCard users see 12 km; the static fallback (for LLM crawlers) says 18 km.

- [ ] **`meta name="color-scheme" content="light"` is hardcoded in `<head>`.** File: `app/index.html` line 6. The ThemeToggle defaults to light and ignores `prefers-color-scheme`. A user whose OS is set to dark will see a flash of light theme on load before React hydrates and reads localStorage. The meta tag reinforces this but also suppresses the browser's native dark-mode scroll bar and form control theming even for users who would prefer dark.
